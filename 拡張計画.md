### アーキテクチャ評価

**良い点**:
- MVVM + Clean Architecture の3層分離が明確
- Hilt によるDI設計が一貫
- Flow/StateFlow による反応的なデータフロー
- 非エンジニアユーザー向けの親しみやすいUI/UXコピー

**改善すべき点**:
- `StartScanningUseCase` / `StopScanningUseCase` が定義済みだが未使用（`MainViewModel` が `ScanServiceController` を直接呼んでいる）
- `collectAsState()` が使われているが `collectAsStateWithLifecycle()` が望ましい
- `isScanning` 状態が DataStore 管理のみで、サービス終了（OS kill）時に不整合が起きうる
- テストコードが Example のみで実質ゼロ

## 既知の不具合・技術的負債

### P0（致命的 — リリース前に必須修正）

| ID | 内容 | 影響 |
|----|------|------|
| B-01 | スキャン開始前に権限チェックがない（`MainViewModel.startScanning`） | 権限未付与でサービス起動 → クラッシュ |
| B-02 | サービスが OS に kill された場合 `isScanning` が `true` のまま残る | UI と実際の動作が乖離 |
| B-03 | `ScanningForegroundService.onDestroy` で `isScanning = false` を DataStore に書き戻していない | 再起動後に不整合 |

### P1（高優先度 — v1.1 で対応）

| ID | 内容 | 影響 |
|----|------|------|
| B-04 | `BootReceiver` が Manifest に宣言済みだが未実装 | 端末再起動後にスキャンが自動復帰しない |
| B-05 | `background_enabled` 設定が実際のサービス制御に反映されていない | 設定OFF でもバックグラウンド動作が止まらない |
| B-06 | `SmartGlassesDetector.manufacturerSpecificData` の取得ロジックで Company ID が `keyAt(0)` ではなくペイロード先頭2バイトから構築されている | 一部デバイスで Company ID 不一致の可能性 |
| B-07 | `release` ビルドで `isMinifyEnabled = false` | APK サイズ肥大・リバースエンジニアリング容易 |

### P2（中優先度 — 技術的負債）

| ID | 内容 |
|----|------|
| D-01 | 未使用の UseCase（`StartScanningUseCase`, `StopScanningUseCase`）の整理 |
| D-02 | `AGENTS.md` / `CLAUDE.md` の「未実装」記載が実装済みの現状と不一致 |
| D-03 | `collectAsState()` → `collectAsStateWithLifecycle()` への統一 |
| D-04 | `ScanningForegroundService` の `CoroutineScope` が `Dispatchers.Default` で固定 |
| D-05 | `StatusCard.kt`, `ScanButton.kt` が存在するが `MainScreen` 内で直接実装されており未使用 |

## まとめ

現在のアプリは **コア機能（BLE スキャン → 検出 → 通知 → 履歴保存）が一通り実装済み** で、Clean Architecture に基づいた堅実な設計がされている。最優先は **Phase 1（安定化）** で、権限チェックの欠如やサービス状態の不整合といった致命的バグを解消し、テスト基盤を整えること。その後、**Phase 2（検出精度）** で Apple/Google Company ID の誤検出問題に対処し、実用的な検出品質を確保する。Phase 3 以降は、ユーザーの声を聞きながら段階的に機能を拡張していく方針が推奨される。
